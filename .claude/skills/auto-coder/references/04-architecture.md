## 4.5 部署拓扑

### 4.5.1 部署分期策略

鉴于当前物理硬件为单节点（AMD Ryzen AI 395 平台）且团队规模固定（20 人），为最大化将系统资源倾斜给 LLM 推理与向量检索，本系统采用**分阶段部署策略**：

* **Phase 1 (单机生产期)**：直接采用 Docker Compose 配合 Systemd 作为生产环境部署标准，实现资源零单点开销（省去 K8s 控制面约 2-4GB 的内存占用）。
* **Phase X (未来扩展期)**：系统架构已原生适配 Kubernetes。若未来业务扩张需迁移至多节点物理机房或公有云，可无缝启用预置的 Helm Charts 方案。

---

### 4.5.2 CPU 核心亲和性

```
Core 0-3  ──→  LM Studio (LLM 推理, GPU/CPU 调度)
Core 4-5  ──→  BGE-M3 Embedding Service (batch 编码)
Core 6    ──→  bce-reranker Service (低频 on-demand)
Core 7-9  ──→  Milvus (HNSW 检索, CPU 密集)
Core 10-11 ──→ PostgreSQL (SQL 查询/写入)
Core 12   ──→  Redis (内存操作, 轻量)
Core 13-15 ──→ Celery Workers (Ingestion 任务)
Shared    ──→  FastAPI Uvicorn Workers (async I/O bound)
```

> **注**: 通过 Docker `--cpuset-cpus` 和 systemd `CPUAffinity` 实现。实际核心数以 AMD Ryzen AI 395 最终规格为准。

### 4.6.2 结构化参数抽取

表格提取后，通过 LLM 将非结构化表格数据转换为标准化参数记录：

```
[LLM Prompt: param_extraction.txt]

Given the following table extracted from {chip_part_number} datasheet, page {page_number}:

{table_content}

Extract ALL electrical/timing/thermal parameters as structured JSON:
{
  "parameters": [
    {
      "name": "VCC",
      "category": "electrical",
      "min": 2.7,
      "typ": 3.3,
      "max": 3.6,
      "unit": "V",
      "condition": "Normal operating conditions",
      "source_table": "Table 5.3.1 General operating conditions"
    },
    ...
  ]
}

Rules:
- Preserve exact values from the table, do not infer missing values
- Categorize as: electrical | timing | thermal | absolute_max
- Include measurement conditions verbatim
- If a cell says "—" or is empty, use null
```

抽取结果直接写入 PostgreSQL `chip_parameters` 表，供 SQL 对比查询使用。

### 4.6.4 Celery 异步任务链

完整 Ingestion 任务链包含 10 个阶段：

1. `download_document` — 下载 PDF 文档（max_retries=3）
2. `validate_and_dedup` — SHA256 去重检查
3. `extract_text` — pdfplumber 全文本提取
4. `extract_tables` — 三级表格提取（pdfplumber → Camelot → PaddleOCR）
5. `extract_structured_params` — LLM 结构化参数抽取（time_limit=120s）
6. `chunk_text` — Datasheet 感知分片
7. `embed_chunks` — BGE-M3 Embedding（调用 :8001 微服务）
8. `store_vectors` — Milvus Upsert（dense + sparse 向量）
9. `store_metadata` — PostgreSQL 元数据写入
10. `sync_knowledge_graph` — Kùzu 图谱同步（v3.0 新增）

```python
# 完整 Ingestion 任务链
def create_ingestion_chain(url: str, manufacturer: str, user_id: int):
    return chain(
        download_document.s(url, manufacturer),
        validate_and_dedup.s(),
        extract_text.s(),
        extract_tables.s(),
        extract_structured_params.s(),
        chunk_text.s(),
        embed_chunks.s(),
        store_vectors.s(),
        store_metadata.s(),
        sync_knowledge_graph.s(),
    ) | notify_completion.si(user_id)
```

### 4.6.5 数据采集三路

**① Playwright 定时爬虫** — Celery Beat 每日凌晨 2:00 执行，抓取 ST/TI/NXP 官网 Datasheet 更新。反爬策略：随机延迟 2-5s、轮换 UA、单次最多 50 PDF、失败指数退避重试。

**② Watchdog 内网目录监听** — 监听 FileCreatedEvent/FileModifiedEvent，仅处理 .pdf，防抖 5s，作为 FastAPI 后台任务启动。

**③ 手动上传** — `POST /api/v1/documents/upload`，支持 PDF，返回 Celery task_id，可通过 `GET /api/v1/tasks/{task_id}` 轮询进度。

核心类：

```python
@dataclass
class AgentConfig:
    max_iterations: int = 5
    max_total_tokens: int = 8192
    parallel_tool_calls: bool = True
    temperature: float = 0.1        # 主推理模型参数
    tool_timeout: float = 30.0
    llm_role: str = "primary"       # "primary" | "router"; 默认使用主推理模型

class AgentOrchestrator:
    """ReAct Agent Orchestrator — v3.0 核心编排引擎"""

    async def run(self, query, conversation_history, session_context, trace) -> AgentResult:
        messages = self.prompt_builder.build(query, history, available_tools, session_context)
        budget = TokenBudget(self.config.max_total_tokens)

        for iteration in range(self.config.max_iterations):
            response = await self.llm.chat_completion(
                messages=messages,
                tools=self.tools.get_openai_tools_schema(),
                tool_choice="auto",
            )
            budget.consume(response.usage.total_tokens)

            if not response.tool_calls:
                return AgentResult(content=response.content, ...)

            tool_calls = [ToolCall(tc.function.name, tc.function.arguments, tc.id)
                         for tc in response.tool_calls]
            observations = await self._execute_tools(tool_calls, trace)

            for obs in observations:
                obs["result"] = self.guardrails.sanitize_tool_output(obs["result"])

            # 追加 Tool 结果到 messages，继续下一轮
            messages.append(response.to_message())
            for obs in observations:
                messages.append({"role": "tool", "tool_call_id": obs["call_id"],
                                "content": str(obs["result"])})

            if budget.exhausted:
                break

        # 超限强制生成最终答案
        messages.append({"role": "system", "content": "请基于已收集的信息生成最终答案。"})
        final = await self.llm.chat_completion(messages=messages)
        return AgentResult(content=final.content, budget_exhausted=True, ...)
```

### 4.9.2 对话式 RAG 管线

**Tool**: `rag_search` | **Pipeline**: `ConversationalRAGPipeline` | **文件**: `src/pipelines/rag_pipeline.py`
**Prompt 模板**: `config/prompts/rag_answer.txt`

**输入**: `ProcessedQuery.rewritten_query`, `ProcessedQuery.filters` (part_number, doc_type, collection)
**输出**: `PipelineResult.content` (LLM 生成的答案), `PipelineResult.citations` (引用列表), `PipelineResult.chunks` (原始检索块)

```python
class ConversationalRAGPipeline(BasePipeline):
    async def execute(self, query: ProcessedQuery, trace: TraceContext) -> PipelineResult:
        # Step 1: Milvus Hybrid Search (Dense + Sparse, RRF k=60)
        vector_results = await self.milvus_store.hybrid_search(
            query=query.rewritten_query, top_k=20,
            filters={"part_number": query.filters.get("part_number"),
                     "doc_type": query.filters.get("doc_type")},
        )
        trace.record_stage("hybrid_search", {"count": len(vector_results)})

        # Step 2: Rerank (bce-reranker, top_k=10)
        reranked = await self.reranker.rerank(query.rewritten_query, vector_results, top_k=10)
        # 降级: Reranker 不可用时直接使用 RRF 排序结果
        if reranked is None:
            reranked = vector_results[:10]
        trace.record_stage("rerank", {"count": len(reranked)})

        # Step 3: 知识库心得补充 (knowledge_notes collection)
        knowledge_hits = await self.milvus_store.hybrid_search(
            query=query.rewritten_query, top_k=3, collection="knowledge_notes"
        )

        # Step 4: LLM Generation (prompt 模板 + context + history)
        context = self._build_context(reranked, knowledge_hits)
        prompt = self.prompt_loader.load("rag_answer.txt", context=context, query=query.original_query)
        answer = await self.llm.chat_completion(messages=[
            {"role": "system", "content": prompt},
            *query.conversation_history,
            {"role": "user", "content": query.original_query}
        ])
        # 降级: LLM 不可用时返回原始 chunk 拼接
        if answer is None:
            return PipelineResult(content=self._fallback_concat(reranked), degraded=True)

        # Step 5: Citation Building
        citations = self._extract_citations(answer.content, reranked)
        return PipelineResult(content=answer.content, citations=citations, chunks=reranked)
```

### 4.9.3 芯片对比管线

**Tool**: `chip_compare` | **Pipeline**: `ChipComparisonPipeline` | **文件**: `src/pipelines/chip_comparison_pipeline.py`
**Prompt 模板**: `config/prompts/chip_comparison.txt`

**输入**: `ProcessedQuery.entities` (part_numbers: list[str], compare_dimensions: list[str])
**输出**: `PipelineResult.content` (LLM 差异分析), `PipelineResult.data` (对比表 dict)

```python
class ChipComparisonPipeline(BasePipeline):
    async def execute(self, query: ProcessedQuery, trace: TraceContext) -> PipelineResult:
        part_numbers = query.entities["part_numbers"]  # 至少 2 个

        # Step 1: SQL 查询每颗芯片的结构化参数
        chip_params = {}
        for pn in part_numbers:
            params = await self.pg.fetch("""
                SELECT cp.parameter_name, cp.parameter_category, cp.min_value, cp.typ_value,
                       cp.max_value, cp.unit, c.status, c.package, c.manufacturer
                FROM chip_parameters cp JOIN chips c ON cp.chip_id = c.id
                WHERE c.part_number = $1
                ORDER BY cp.parameter_category, cp.parameter_name
            """, pn)
            chip_params[pn] = params
        trace.record_stage("sql_query", {"chips": len(chip_params)})

        # Step 2: 参数对齐 — 构建可比较的参数矩阵
        compare_table = self._align_parameters(chip_params, query.entities.get("compare_dimensions"))

        # Step 3: LLM 差异分析
        prompt = self.prompt_loader.load("chip_comparison.txt",
            compare_table=compare_table, part_numbers=part_numbers)
        analysis = await self.llm.chat_completion(messages=[
            {"role": "system", "content": prompt},
            {"role": "user", "content": query.original_query}
        ])
        # 降级: LLM 不可用时仅返回对比表
        if analysis is None:
            return PipelineResult(content="LLM 不可用，仅返回结构化对比表。", data=compare_table, degraded=True)

        # Step 4: 向量检索补充设计注意事项 (App Note / Errata)
```

```python
class BOMReviewPipeline(BasePipeline):
    async def execute(self, query: ProcessedQuery, trace: TraceContext) -> PipelineResult:
        # Step 1: Excel 解析 — 提取 part_number 列表
        bom_items = self._parse_excel(query.entities["bom_file_path"])
        trace.record_stage("parse_excel", {"items": len(bom_items)})

        # Step 2: 型号匹配 — SQL 批量查询 chips 表
        review_results = []
        for item in bom_items:
            chip = await self.pg.fetchrow(
                "SELECT id, part_number, status, manufacturer FROM chips WHERE part_number = $1",
                item["part_number"])

            result = {"row": item["row"], "part_number": item["part_number"],
                      "matched": chip is not None}

            if chip:
                # Step 3: EOL/NRND 检查
                result["eol_flag"] = chip["status"] in ("EOL", "NRND", "obsolete")

                # Step 4: 参数冲突检测 (如 BOM 指定电压范围与 Datasheet 不符)
                if item.get("specifications"):
                    conflicts = await self._check_param_conflicts(chip["id"], item["specifications"])
                    result["param_conflicts"] = conflicts

                # Step 5: 替代推荐 (EOL 或存在严重勘误时)
                if result["eol_flag"]:
                    alts = await self.pg.fetch("""
                        SELECT c.part_number, c.manufacturer, ca.compat_type, ca.compat_score
                        FROM chip_alternatives ca JOIN chips c ON ca.alt_id = c.id
                        WHERE ca.original_id = $1 AND c.status = 'active'
                        ORDER BY ca.compat_score DESC LIMIT 3
                    """, chip["id"])
                    result["alternatives"] = alts

            review_results.append(result)
        trace.record_stage("review", {"flagged": sum(1 for r in review_results if r.get("eol_flag"))})

        # Step 6: LLM 审查摘要
        prompt = self.prompt_loader.load("bom_review.txt", results=review_results)
        summary = await self.llm.chat_completion(messages=[
            {"role": "system", "content": prompt},
            {"role": "user", "content": f"请审查以下 BOM，共 {len(bom_items)} 个器件"}
        ])

        # Step 7: 持久化 (bom_records + bom_items 表)
        bom_id = await self._persist_review(query, review_results, summary)

        return PipelineResult(content=summary.content if summary else "LLM 不可用", data=review_results)
```

```python
class ChipSelectionPipeline(BasePipeline):
    async def execute(self, query: ProcessedQuery, trace: TraceContext) -> PipelineResult:
        filters = query.entities

        # Step 1: SQL 结构化过滤 — 缩小候选集
        candidates = await self.pg.fetch("""
            SELECT DISTINCT c.id, c.part_number, c.manufacturer, c.category,
                   c.package, c.pin_count, c.status
            FROM chips c
            LEFT JOIN chip_parameters cp ON cp.chip_id = c.id
            WHERE c.status = 'active'
              AND ($1::varchar IS NULL OR c.category = $1)
              AND ($2::varchar IS NULL OR c.package = $2)
              AND ($3::int IS NULL OR c.pin_count = $3)
            ORDER BY c.manufacturer, c.part_number
            LIMIT 50
        """, filters.get("category"), filters.get("package"), filters.get("pin_count"))
        trace.record_stage("sql_filter", {"candidates": len(candidates)})

        # Step 2: 向量语义排序 — 用 Rerank 按需求描述相关性排序
        if query.rewritten_query and len(candidates) > 5:
            candidate_texts = [f"{c['part_number']} {c['manufacturer']} {c['category']}" for c in candidates]
            reranked = await self.reranker.rerank(query.rewritten_query, candidate_texts, top_k=10)
            candidates = [candidates[i] for i in reranked.indices] if reranked else candidates[:10]

        # Step 3: 国产替代匹配
        if filters.get("prefer_domestic"):
            for c in candidates:
                domestic_alts = await self.pg.fetch("""
                    SELECT alt.part_number, alt.manufacturer, ca.compat_score
                    FROM chip_alternatives ca JOIN chips alt ON ca.alt_id = alt.id
                    WHERE ca.original_id = $1 AND ca.compat_type = 'domestic_alt' AND alt.status = 'active'
                """, c["id"])
                c["domestic_alternatives"] = domestic_alts

        # Step 4: LLM 推荐理由
        prompt = self.prompt_loader.load("chip_selection.txt",
            candidates=candidates, requirements=query.original_query)
        recommendation = await self.llm.chat_completion(messages=[
            {"role": "system", "content": prompt},
            {"role": "user", "content": query.original_query}
        ])
        # 降级: LLM 不可用时仅返回候选列表
        if recommendation is None:
            return PipelineResult(data=candidates, degraded=True)

        return PipelineResult(content=recommendation.content, data=candidates)
```

### 4.10.2 QueryRewriter

将多轮对话中的代词/省略引用改写为完整独立查询。使用轻量路由模型 (router)，仅消耗 ~50 tokens，延迟 ~30ms。

```python
class QueryRewriter:
    REWRITE_PROMPT = """根据对话历史，将用户最新消息改写为完整独立查询，解析所有代词。
    对话历史: {history}
    用户最新消息: {current_query}
    改写后的独立查询:"""

    async def rewrite(self, current_query, history) -> str:
        if not history or not self._needs_rewrite(current_query):
            return current_query
        response = await self.router_llm.generate(prompt, temperature=0, max_tokens=100)
        return response.text.strip()

    def _needs_rewrite(self, query):
        pronouns = ["它", "这个", "那个", "其", "该", "上面", "前面", "这款", "那款",
                     "it", "this", "that", "these", "those", "the same"]
        return any(p in query.lower() for p in pronouns)
```

### 4.11.2 熔断器

```python
class CircuitBreaker:
    """三态熔断器: CLOSED → OPEN → HALF_OPEN → CLOSED"""
```

### 4.11.3 Tenacity 重试

协作流程：

```
请求进入 → CircuitBreaker.call()
    ├─ OPEN → 直接抛出 CircuitBreakerOpen (快速失败)
    ├─ CLOSED/HALF_OPEN → Tenacity 包裹的实际调用
    │       ├─ 成功 → 返回结果
    │       └─ 重试耗尽仍失败 → CircuitBreaker 计数 → 超阈值切 OPEN
    └─ 返回 / 抛出
```

代码示例：

```python
from tenacity import retry, stop_after_attempt, wait_exponential, retry_if_exception_type

def llm_retry(func):
    """LLM 调用: 2 次重试, 2-15s 指数退避"""
    return retry(
        stop=stop_after_attempt(3),
        wait=wait_exponential(multiplier=1, min=2, max=15),
        retry=retry_if_exception_type((ConnectionError, TimeoutError)),
        reraise=True,
    )(func)
```

### 4.11.4 降级策略矩阵

| 故障组件 | 降级行为 | 用户感知 |
|----------|---------|---------|
| **LLM** | 返回 GPTCache → 仅返回原始 chunks → 提示"AI 摘要暂不可用" | 有感知, 核心数据仍可用 |
| **BGE-M3** | 退回 PostgreSQL `ILIKE` + `ts_vector` 关键词搜索 | 检索质量下降 |
| **bce-reranker** | 跳过 Rerank, 直接使用 RRF 融合结果 | 排序可能不够精准 |
| **Milvus** | 仅使用 PostgreSQL 结构化查询 | 语义检索不可用 |
| **PostgreSQL** | 对话 RAG 仍可工作 (纯 Milvus) | 结构化功能不可用 |
| **Redis** | 禁用缓存; 会话退回内存; 限流退回进程内计数器 | 透明, LLM 负载增加 |
| **Celery** | 上传排队, 在线查询不受影响 | 新文档入库延迟 |
| **Kùzu** | graph_query 返回空 + 警告; Agent 退回 SQL+向量检索 | 图谱功能降级 |

### 4.11.5 20 人并发保障

```
20 人团队典型使用: 高峰 10-15 人在线, 峰值 QPS ~0.5
主推理模型瓶颈: 有效并发 2 路 (35B 级), ~0.7 req/s (无缓存)
路由模型: 有效并发 10+ 路, 查询改写/意图分类延迟 < 50ms
优化: 路由模型处理简单请求 (改写/分类)，仅复杂推理进入主模型队列
结论: GPTCache 命中率 > 30% + 路由模型分流时, 单机可支撑 20 人并发
```

```
              请求进入
                 │
    ┌────────────▼────────────┐
    │  Layer 1: GPTCache      │  拦截 30-50% 重复查询 (p99 < 50ms)
    └────────────┬────────────┘
                 │ (Cache MISS)
    ┌────────────▼────────────┐
    │  Layer 2: Rate Limiter  │  30 req/min/user, 保护公平使用
    └────────────┬────────────┘
    ┌────────────▼────────────┐
    │  Layer 3: LLM Semaphore │  主模型并发 ≤ 2, 路由模型并发 ≤ 10
    └────────────┬────────────┘

    └─────────────────────────┘
```

| **user** | ✅ | ✅ | ✅ | ✅ (自己的) | ❌ | ❌ |
| **viewer** | ✅ | ❌ | ✅ (只读) | ❌ | ❌ | ❌ |

```python
def require_role(*roles: str):
    """RBAC 装饰器, 用于路由级权限控制"""
    def decorator(func):
        @wraps(func)
        async def wrapper(*args, current_user=Depends(get_current_user), **kwargs):
            if current_user.role not in roles:
                raise HTTPException(status_code=403, detail="权限不足")
            return await func(*args, current_user=current_user, **kwargs)
        return wrapper
    return decorator
```

### 4.12.4 安全措施总表

| 安全层 | 措施 | 实现方式 |
|--------|------|---------|
| **传输安全** | HTTPS | Nginx / Caddy TLS 终端 |
| **认证** | SSO/OIDC + 内部 JWT (RS256) | OIDC Middleware + JWKS 验证 |
| **授权** | RBAC (3 角色) + IdP 组映射 | 路由级装饰器 + JIT Provisioning |
| **限流** | Token Bucket | Redis 计数器 |
| **输入校验** | Pydantic 严格模式 | Request Schema 验证 |
| **SQL 注入** | 参数化查询 | asyncpg 绑定参数 |
| **文件上传** | 类型白名单 + 大小限制 | .pdf / .xlsx, max 100MB |
| **CORS** | 白名单域名 | FastAPI CORSMiddleware |
| **日志脱敏** | 密码/Token 不入日志 | 自定义 JSONFormatter |
| **CSRF** | OAuth2 state + OIDC nonce | Redis 一次性验证 |
| **Token 安全** | HttpOnly + Secure Cookie | 防 XSS 窃取 |
| **Refresh 轮转** | Refresh Token Rotation | 旧 token 立即失效 |
| **内部通信** | 明文 (Phase 1) / mTLS (Phase X) | Phase 1 所有服务绑定 `127.0.0.1` 或 Docker 内部网络，loopback 无需加密；Phase X 通过 K8s NetworkPolicy + 服务网格 mTLS |
| **密钥管理** | Env vars (Phase 1) / K8s Secrets (Phase X) | `.env` 文件 (gitignored) + Docker Compose `env_file` |

### 4.12.5 密钥管理策略

**Phase 1（单机部署）**:
- 所有敏感配置存储在项目根目录 `.env` 文件中（已加入 `.gitignore`）
- `docker-compose.yml` 通过 `env_file: .env` 引用
- 提供 `.env.example` 模板（不含真实密码）供团队参考
- 密码最低要求：16 字符，建议使用 `openssl rand -base64 24` 生成

**JWT 密钥轮换**:
- RS256 PEM 密钥对，建议每 90 天轮换
- 轮换期间保留旧公钥至所有已签发 Token 过期（max TTL 7 天）
- 开发环境可降级为 HS256 单密钥

**Phase X（K8s 集群）**:
- 迁移至 K8s Secrets + External Secrets Operator（如 AWS Secrets Manager / HashiCorp Vault）
- 通过 Sealed Secrets 加密后可安全提交到 Git

## 4.13 云原生架构储备 (Future Evolution)

> **架构储备声明**：以下 Kubernetes 与 Helm 配置为系统横向扩展（Scale-out）的储备蓝图。在 Phase 1 的单机部署阶段无需启用，仅当系统需跨越单台物理机限制时激活。

### 4.13.0 迁移触发条件

从 Phase 1（单机 Docker Compose）迁移至 Phase X（K8s 集群）的决策矩阵：

| 触发条件 | 阈值 | 依据 |
|----------|------|------|
| **并发用户数** | 持续 > 50 人 | 单机主模型并发 ≤ 2 成为瓶颈 (§4.11.5) |
| **Milvus 数据量** | > 1M chunks | Standalone 性能天花板 (§4.11.6) |
| **可用性 SLA** | > 99.5% | 需 HA 副本，单节点无法保证 |
| **第二台物理机** | 可用 | 硬件前提：需额外计算节点 |
| **团队规模** | > 50 人 | 多部门隔离需求，需命名空间级别租户 |

满足**任意两项**即建议启动 Phase X 迁移评估。
